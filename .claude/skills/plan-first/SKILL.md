# خطّط أولاً ثم نفّذ

## الترتيب الإلزامي
1. **اقرأ قبل أن تحكم:** افتح الملفات الفعلية وتتبّع المسار من المدخل حتى الأثر (DB/API/الإخراج). لا تفترض من الاسم أو الذاكرة.
2. **تحقّق من الواقع لا الوصف:** إن قال الطلب "X موجود/مفعّل"، أثبته بالكود. ميّز "موجود كملف" عن "مُطبَّق فعلاً" (مثال: migration في ملف لكنها لا تعمل عند النشر).
3. **اكشف التعارضات مبكراً:** إن تعارض طلب المستخدم مع تعليمات النظام/الإعداد، أو مع حالة الكود — أشِر إليه قبل التنفيذ. تعليمات المستخدم الصريحة تسبق الافتراضات.
4. **قرارات المستخدم تُسأل، لا تُفترض:** عند مفترق حقيقي يغيّر النتيجة (وخاصة على بيانات حية / فعل صعب التراجع)، اعرض الخيارات بمفاضلة وتوصية واسأل. أما القرارات ذات الافتراض المعقول فاتخذها وأعلِنها.
5. **خطة بأولويات:** قبل الكتابة، أخرِج خطة قصيرة: الخطوات، الملفات المتأثرة، المخاطر، ومعيار النجاح. رتّب حسب (القيمة × الأمان).

## أثناء التنفيذ
- نفّذ على دفعات صغيرة قابلة للتحقق؛ تحقّق من البناء بعد كل دفعة لا في النهاية فقط.
- فضّل أصغر تغيير يحقق الهدف؛ طابِق أسلوب الكود المحيط.
- إن كانت المهمة كبيرة/محفوفة بالمخاطر، أنجز الجزء الآمن عالي القيمة بإتقان ووثّق المتبقّي بدل تغيير ضخم متهوّر.

## قبل الإعلان عن الإنجاز
- شغّل التحقق الفعلي (اختبار/بناء/تشغيل) وانقل النتيجة بصدق: إن فشل شيء قُلْه، وإن تُخطّيت خطوة فاذكرها.
- لا تقل "تم والتحقق ناجح" إلا بعد ملاحظة النجاح فعلاً.

## Git
الفرع الذي يحدّده المستخدم صراحةً. لا Pull Request إلا بطلب صريح.
