# دليل تدقيق ما قبل الإنتاج — Masarat ERP
### نسخة احترافية: برومبتات جاهزة + خطة معيارية (الأولوية للمحاسبة)

---

## 0) شروط قبل التشغيل (لا غنى عنها)

- **ارفع الكود فعلياً** (مجلد المشروع أو zip لآخر ديبلوي). بدون مصدر، أي اكتشاف = تخمين مرفوض.
- **لغة التقرير:** العربية. أما المسارات ورموز الكود وأسماء الجداول/الدوال فتبقى بالإنجليزية كما هي في الكود.
- **بلا وكلاء فرعيين:** الموديل نفسه يقرأ الكود ويحكم وينفّذ مباشرة. لا تفويض لـ subagents — لأن قاعدة الدليل تتطلب أن يكون الحاكم هو من قرأ السطر بنفسه، والوكيل الفرعي يُعيد ملخصاً لا السطر.
- **التنفيذ مرحلي، طوراً بطور:** أنجز كل طور في جلسة مستقلة قدر الإمكان (لإبقاء السياق نظيفاً وعميقاً)، سلّم تقريره، وراجعه قبل بدء الطور التالي. ابدأ بالمحاسبة.

## 1) المبدأ المنهجي — يُلصق حرفياً في رأس كل طور

```
أنت مدقق نظم محاسبية ومالية. منهجك إلزامي:

[قاعدة الدليل] كل اكتشاف يجب أن يحتوي على الحقول الخمسة، وإلا فهو مرفوض:
  1. الملف:السطر (path:line)
  2. مقتطف الكود الفعلي المنسوخ من الملف (لا إعادة صياغة)
  3. السيناريو الواقعي: خطوات يومية في الوكالة تُنتج الضرر
  4. الأثر المالي: هل يُسبّب تحريفاً جوهرياً في القوائم / إقراراً ضريبياً خاطئاً / فقدان أموال؟
  5. الإصلاح المقترح: أسطر كود محددة أو تغيير schema محدد

[قاعدة الغياب] إذا ادّعيت أن شيئاً "غير موجود" (مثل: لا يوجد قفل فترة)،
  يجب أن تُرفق أمر البحث الذي شغّلته (grep/glob) ونتيجته الفارغة.
  لا تؤكد الغياب بلا بحث موثّق. ادعاءات الغياب أخطر مصدر للأخطاء.

[الحياد] النظام بريء حتى تُثبت إدانته بمقتطف كود، ومذنب حتى تتحقق من براءته بالقراءة.
  لا "يُحتمل"، لا "غالباً". إن لم تقرأ السطر فلا تكتب الاكتشاف.

[الثقة] لكل اكتشاف: درجة ثقة (مؤكد / مرجّح / يحتاج تحقق يدوي) وكيفية إعادة إنتاجه.

[الاحترافية الوظيفية + صدق الواجهة] لكل ميزة لها واجهة: تتبّعها حتى النهاية، لا تكتفِ بأنها "ظاهرة":
  - **قاعدة الميزة الميتة:** زر/مفتاح/تبويب/نموذج يغيّر حالة محلية فقط (state) ولا يستدعي API ولا
    يُحفظ في DB ولا يؤثر بعد refresh = "ميزة وهمية" — عيب احترافي يُسجَّل بحقوله الخمسة. أرفِق مسار
    الكتابة (onClick → API → DB) أو أثبت غيابه (قاعدة الغياب).
  - **صدق الواجهة:** كل نص يَعِد بأثر ("يؤثر فوراً"، "يُفعّل لكل المستخدمين") يجب أن يطابق السلوك
    الفعلي؛ النص الكاذب عيب.
  - **الجودة الاحترافية والتصميمية:** حلّ النظام يجب أن يضاهي الأنظمة المرجعية (تحكّم دقيق لا حزمة
    جامدة)، والمصطلح العربي صحيح نحوياً (رفع/نصب/جر) ومعتمد، والاتساق البصري وحالات (فراغ/تحميل/خطأ)
    موجودة، و**التجاوب على الهاتف والكمبيوتر معاً** (لا تصميم لجهاز يُهمل الآخر). هذه البنود "شكلية"
    بالمظهر لكنها تمسّ الاحترافية وكفاءة العمل الحقيقي — لا تتجاهلها لأنها لا تُعطّل قيداً مالياً.
    (التفصيل في الطور 6.)

التصنيف: Critical (يمنع الإطلاق) / High (قبل أول عميل) / Medium / Low

[تنبيه التبديل — إلزامي] بعد تسليم تقرير الطور، اختم بسطر بارز ثابت:
  "⏹️ انتهى الطور [رقمه]. قبل الطور التالي بدّل النموذج إلى [النموذج] والجهد إلى [المستوى]
   حسب جدول النموذج/الجهد، وافتح جلسة جديدة."
  لا تبدأ الطور التالي في نفس الجلسة. انتظر تبديل المستخدم للنموذج.
```

## 2) الطور 0 — الاستطلاع وخريطة النظام (إلزامي قبل أي تدقيق)
`🧠 Opus 4.8 · جهد xhigh`

```
المهمة: ارسم خريطة النظام. لا تُصدر أحكاماً بعد. أخرِج فقط الخريطة.

1. حدّد المكدّس التقني: الإطار، قاعدة البيانات، ORM، إدارة الجلسات/JWT.
2. حدّد ملفات الـ schema / migrations. استخرج تعريفات الجداول:
   accounts, journals, journal_lines, invoices, invoice_lines,
   payments, bookings, customers, suppliers, tax_*, period_locks.
   لكل جدول: الأعمدة، الأنواع، المفاتيح، القيود (FK/UNIQUE/CHECK)، الفهارس.
3. أرقام الأعمدة المالية: ما نوع تخزين المبالغ؟ (هللات كـ integer ✅ / decimal ✅ / float ❌).
4. ابحث عن كل نقطة تُنشئ قيداً محاسبياً:
   grep على إدراج journal_lines / دالة postJournal / createJournalEntry وما شابه.
   أخرِج قائمة بكل المستدعين (callers) مع مسار:سطر.
5. أين دليل الحسابات (chart of accounts)؟ هل الأرقام (1110/1120/2000/2200/2300/3201/4xxx/5000) ثابتة في الكود أم في الجدول؟
6. ارسم تدفق البيانات: عرض سعر → حجز → فاتورة → دفع → تقرير.
   لكل خطوة: ما الجداول التي تُكتب، وهل تُنشأ قيود؟
7. [حقيقة هذا المستودع — لا تدقّق SQL ميتاً] المُطبِّق الفعلي للـ migrations هو
   apps/web/src/instrumentation.ts (مصفوفة SQL idempotent تعمل عند كل إقلاع). ملفات
   apps/web/drizzle/*.sql توثيقية فقط ولا تُطبَّق (build لا يشغّل migrate). ⇒ عند فحص أي
   عمود/قيد/فهرس، اقرأ instrumentation.ts لا drizzle/*.sql، وإلا دقّقت جدولاً لا وجود له إنتاجياً.
8. [فحص فجوة الـ schema — ميكانيكي وعالي القيمة] قارِن الأعمدة المُشار إليها في الكود
   (lib/schema.ts + الاستعلامات) مقابل الأعمدة المُنشأة فعلاً في instrumentation.ts.
   أي عمود يقرأه/يكتبه الكود وغير منشأ = عطل إنتاجي صامت يفشل وقت التشغيل لا البناء
   (حدث فعلاً: buyer_vat_number أوقف إصدار الفواتير بـ 500). أخرِج قائمة الفجوات إن وُجدت.

المخرَج: جدول جداول + قائمة نقاط إنشاء القيد + مخطط تدفق نصي + قائمة فجوات schema. مرجع لكل الأطوار التالية.
```

🔄 **بعد تسليم تقرير الاستطلاع:** بدّل إلى **Opus 4.8 · Max** قبل بدء الطور 1 المحاسبي (جلسة جديدة).

## 3) الطور 1 — التدقيق المحاسبي العميق (أولويتك القصوى)
`🧠 Opus 4.8 · جهد Max — أعلى مستوى، والبطء هنا مقصود`

> اقرأه أنت (الموديل) مباشرة بشكل متسلسل تتبع فيه الخيط. مقسّم إلى 4 وحدات. سلّم تقرير كل وحدة قبل التالية.

### الوحدة 1-أ: العميل إلى التحصيل (Order-to-Cash) — من أول تواصل

```
تتبّع الرحلة الكاملة لكل نوع خدمة (طيران/فندق/تأشيرة/باقة/نقل):
  عرض سعر → حجز → فاتورة → دفع → تحصيل → استرداد/إلغاء.

تحقق نقطة بنقطة بالكود:

[الاعتراف بالإيراد — الأخطر] هل النظام يميّز "الوكيل مقابل الأصيل" (IFRS 15)؟
  - تذكرة طيران (BSP/IATA): الوكالة وكيل → الإيراد = العمولة فقط، لا إجمالي التذكرة.
    افحص: هل القيد يقيّد إجمالي التذكرة كإيراد 4xxx؟ هذا تحريف جوهري + VAT خاطئ.
  - باقة سياحية تبيعها الوكالة كأصيل → الإيراد إجمالي. تأكد أن الكود يفرّق بين الحالتين.
  أعطِ مقتطف الكود الذي يحدد مبلغ الإيراد لكل نوع.

[العربون/الدفعة المقدمة] دفعة قبل إصدار الخدمة:
  يجب DR نقدية 1110 / CR التزام دفعات مقدمة (مطلوب، ليس إيراد). متى يُعترف بالإيراد؟
  افحص هل العربون يُسجَّل خطأً كإيراد فوري.

[الفوترة] إصدار الفاتورة يجب أن يُنتج:
  DR ذمم مدينة 1120/1230 / CR إيراد 4xxx / CR ضريبة مستحقة 2200.
  تحقق أن المبالغ بالهللات وأن الضريبة محسوبة على الوعاء الصحيح.

[الدفع] DR نقدية/بنك 1110 / CR ذمم. افحص الدفع الجزئي والزائد (overpayment).

[العملات] فاتورة بـ USD ودفع بـ SAR: هل يُحتسب فرق صرف محقّق (مكسب/خسارة 4xxx/5xxx)؟
  أم يُهمل ويختل التطابق؟ أرفِق مقتطف منطق الصرف.

[الإلغاء/الاسترداد] لكل سيناريو، تتبّع القيد العكسي:
  هل يعكس الإيراد والضريبة معاً؟ هل يُنشئ إشعاراً دائناً 381؟ هل يحترم غرامة الإلغاء؟

لكل اكتشاف طبّق الحقول الخمسة.
```

### الوحدة 1-ب: المورد إلى الدفع (Procure-to-Pay)

```
تتبّع: فاتورة المورد (طيران عبر BSP، فنادق، مزوّد تأشيرات) → استحقاق → دفع.
  - تكلفة المبيعات 5000 / دائنة 2000، وضريبة المدخلات (1230/قابلة للاسترداد).
  - تسوية BSP الدورية: هل توجد آلية مطابقة بين كشف BSP ودفتر الموردين؟
  - هل تُسجَّل تكلفة التذكرة (في حالة الوكيل) أم صافي العمولة فقط؟ تأكد من الاتساق مع 1-أ.
الحقول الخمسة لكل اكتشاف.
```

### الوحدة 1-ج: القيد إلى التقرير (Record-to-Report) — التطابقات

```
نفّذ اختبارات المطابقة بقراءة منطق التقارير (ولو يدوياً على عيّنة):
  1. ميزان المراجعة = مجموع كل journal_lines (مدين = دائن إجمالاً).
  2. دفتر الذمم المدينة المساعد = رصيد حساب المراقبة 1120/1230 في الأستاذ.
  3. دفتر الموردين المساعد = رصيد 2000.
  4. الإقرار الضريبي = حركة 2200 (مخرجات) و1230 (مدخلات).
  5. قفل الفترة: هل أي قيد بتاريخ داخل فترة مغلقة يُرفض؟ أرفِق مقتطف الفحص أو أثبت غيابه.
لكل تقرير: من أين يجلب الأرقام؟ هل من journal_lines (صحيح) أم من جدول مشتق قد يتباعد (خطر)؟
```

### الوحدة 1-د: سلامة القيد المحاسبي (شاملة)

```
افحص بالكود:
  [التوازن] هل يُرفض أي قيد مجموع مدينه ≠ دائنه؟ عند الكود وعند DB (CHECK)؟
  [الذرّية] هل سطور القيد تُكتب داخل DB transaction واحدة؟ ابحث عن غياب transaction/commit/rollback.
    سيناريو: انقطاع بين السطر الأول والثاني → قيد نصفي يفسد الدفاتر.
  [الذرّية تحت التزامن — Serverless · الأهم] وجود transaction لا يكفي. كل تحديث لرصيد مالي
    (paidHalalas، رصيد مورد/عميل، مبلغ مسترَد، مخزون) يجب أن يكون UPDATE شرطياً واحداً
    بـ WHERE-guard يُعيد الصفوف المتأثرة (RETURNING) — مثال سليم:
      UPDATE invoices SET paid = paid + :amt WHERE (total - paid) >= :amt RETURNING …
      ثم: 0 صفوف ⇒ ارفض (سباق فاز عليه). الاسترداد: WHERE paid >= refund+fee → 409.
    النمط الخطر = read-then-write (SELECT الرصيد ثم UPDATE بقيمة محسوبة في التطبيق): على
    Neon serverless تحت نقرتين متزامنتين = lost update ⇒ دفع/استرداد مزدوج. تحقّق: ابحث عن
    SELECT ثم UPDATE منفصلين على نفس الصف داخل المعاملة، وعن UPDATE بلا WHERE-guard على الرصيد.
  [عدم التكرار Idempotency] ضغط "دفع/فوترة" مرتين أو نقرتان متسارعتان: هل يُنشأ قيدان؟
    ابحث عن مفتاح idempotency أو قيد UNIQUE يمنع التكرار.
  [نوع المال] هل أي حقل مبلغ من نوع float/double/Number عائم؟ (ممنوع — أخطاء تقريب تراكمية).
  [الحصانة] هل يمكن تعديل/حذف قيد مرحّل؟ يجب القفل والتصحيح بقيد عكسي فقط. أرفِق منطق التعديل/الحذف.
  [الترقيم المتسلسل] أرقام الفواتير/القيود متسلسلة بلا فجوات (متطلب ZATCA)؟ افحص توليد الرقم والسباقات.
  [أثر التدقيق] created_by/created_at، ومنع الحذف الصلب (hard delete) للسجلات المالية.
```

🔄 **بعد تسليم تقارير الوحدات أ–د:** الطور 2 يبقى **Opus 4.8 · Max** (تتبّع مالي). ابقَ على Max وافتح جلسة جديدة.

## 4) الطور 2 — دورة حياة الحجوزات (40 سيناريو)
`🧠 Opus 4.8 · جهد Max`

```
لكل نوع خدمة (طيران/فندق/تأشيرة/باقة/نقل) تتبّع: حجز→فوترة→دفع→إلغاء/استرداد/تعديل.
ركّز على الحالات الحدية:
  - دفع جزئي ثم إلغاء (كيف يُحسب المسترد والقيد العكسي؟)
  - إشعار دائن لفاتورة مدفوعة بالكامل
  - تعديل بعد الفوترة (هل يُنشئ إشعار 383 أم يعدّل الفاتورة الأصلية خطأً؟)
  - عمليات متزامنة على نفس الحجز (سباق race condition)
  - أي عملية بتاريخ فترة مغلقة
لكل سيناريو: مسار الكود + القيد الناتج + الحقول الخمسة. هذا الطور يستند لخريطة الطور 0 ويتكامل مع 1-أ.
```

🔄 **بعد تسليم تقرير الحجوزات:** يمكنك النزول إلى **Opus 4.8 · xhigh** (أو Sonnet 4.6 · high للأجزاء الميكانيكية) للطور 3.

## 5) الطور 3 — الضرائب وZATCA
`🧠 Opus 4.8 · xhigh — أو Sonnet 4.6 · high للأجزاء الميكانيكية`

```
افحص بالكود:
  - الفاتورة الضريبية B2B (مع الرقم الضريبي للمشتري) مقابل B2C المبسّطة.
  - الإشعارات الدائنة 381 / المدينة 383: الحقول والربط بالأصل.
  - QR وفق المواصفة، سلسلة PIH/ICV (هل ICV متسلسل بلا فجوة؟ PIH = هاش السابقة فعلاً؟).
  - الإعفاءات وأكواد VATEX (مثل VATEX-SA-34-1) في الأماكن الصحيحة.
  - تطابق الإقرار الضريبي مع دفتر 2200/1230 (يتكامل مع 1-ج).
الحقول الخمسة لكل اكتشاف.
```

🔄 **بعد تسليم تقرير الضرائب:** بدّل إلى **Sonnet 4.6 · high** للطور 4 (مسح أمني عريض وأوفر).

## 6) الطور 4 — العزل بين الوكالات والأمان
`🧠 Sonnet 4.6 · high — ارفع أي بند مشكوك فيه إلى Opus · Max`

```
  - كل استعلام في api/**/route.ts مقيّد بـ agencyId مأخوذ من JWT، لا من body/params/query.
    أرفِق كل مسار يأخذه من مصدر غير موثوق.
  - صلاحيات الأدوار على كل mutation مالي (من يرحّل قيداً؟ من يصدر إشعاراً دائناً؟).
  - IDOR في مسارات [id] والموارد المتداخلة (هل يصل وكيل لمورد وكيل آخر بتغيير id؟).
  - حقن SQL: ابحث عن sql.raw والاستعلامات المبنية بالنصوص.
  - تسريب أسرار/بيانات حساسة في الاستجابات (hash كلمة المرور، مفاتيح، بيانات وكالة أخرى).
الحقول الخمسة لكل اكتشاف.
```

🔄 **بعد تسليم تقرير الأمان:** ابقَ على **Sonnet 4.6 · medium–high** للطور 5.

`🧠 Sonnet 4.6 · medium–high`

```
  - حذف مورد/عميل له ذمم قائمة: هل يُمنع؟ أم يُيتّم القيد؟
  - تعديل فاتورة مرحّلة (يتكامل مع 1-د).

## 7.5) الطور 6 — الاحترافية الوظيفية وجودة التجربة (UX/Design)
`🧠 Opus 4.8 · جهد high–xhigh — تتبّع الميزة حتى الـ backend + حكم على الجودة`

> **منشأ هذا الطور (جلسة 2026-06-21 — صلاحيات المستخدم/الوحدات):** ظهرت عيوب لم يصطدها أي طور مالي/أمني
> لأنها لا تُعطّل قيداً ولا تكسر العزل، لكنها تُسقِط احترافية النظام وكفاءة عمله الحقيقي:
> تبويب "الوحدات" كان يبدّل حالة محلية بلا حفظ (ميزة وهمية)؛ تحذير أصفر يَعِد بأثر غير موجود (نص كاذب)؛
> صلاحيات بحزمة دور جامدة بلا تحكّم دقيق (حل غير احترافي)؛ وتسمية/نحو. **القاعدة المستخلصة:** الميزة لا
> تُعَدّ منجزة لأنها "ظاهرة" — تُعَدّ منجزة حين تؤدي العمل الحقيقي وتطابق الأنظمة الاحترافية.

```
المهمة: دقّق كل ميزة لها واجهة على محورين — هل تعمل فعلاً؟ وهل احترافية؟ لا تُصدر حكماً بلا تتبّع كود.

[قاعدة الميزة الميتة] لكل زر/مفتاح/تبويب/نموذج/فلتر: تتبّع المسار حتى النهاية.
  هل onClick يستدعي API فعلياً؟ هل API يكتب في DB؟ هل الأثر دائم بعد refresh؟
  ما يغيّر حالة React المحلية (setState) فقط = ميزة وهمية تُضلّل المستخدم. أرفِق سلسلة
  (المكوّن:السطر → مسار API → كتابة DB) أو أثبت انقطاعها (قاعدة الغياب: أرفِق grep الفارغ).

[قاعدة صدق الواجهة] كل نص يَعِد بأثر ("يؤثر فوراً"، "يُفعّل لكل المستخدمين", "تم الحفظ") قارنه
  بالسلوك الفعلي في الكود. النص الكاذب أو المُضخِّم عيب يُسجَّل.

[الاحترافية الوظيفية] قارن الحل بالأنظمة المرجعية (Zoho/Odoo/SAP/NetSuite):
  - تحكّم دقيق حيث يلزم (صلاحيات بالأقسام، فلاتر، إعدادات) بدل حزم جامدة بلا خيار.
  - الفرض الحقيقي في الخادم لا الواجهة فقط (مربعات/إخفاء UI = تجميل فوق حارس خلفي؛ لو الحارس
    الخلفي غائب فالميزة غير آمنة وغير احترافية — يتكامل مع الطور 4).
  - تغطية الحالات: الفراغ (empty)، التحميل (loading)، الخطأ (error)، التوافق الرجعي للبيانات القديمة.

[الجودة اللغوية والتصميمية]
  - المصطلح العربي صحيح نحوياً (رفع/نصب/جر؛ مثال: "المستخدمون" كعنوان مرفوع مقابل "إدارة المستخدمين")
    ومطابق للمصطلح المحاسبي/الإداري المعتمد.
  - الاتساق البصري، RTL، أحجام اللمس، إمكانية الوصول، توحيد المكوّنات المتكررة.
  - **التجاوب (Responsive) — إلزامي على الجهازين:** كل شاشة/نموذج/جدول يجب أن يُعرض بكفاءة على
    **الهاتف والكمبيوتر** معاً. افحص نقاط الكسر (breakpoints)، عدم تجاوز المحتوى أو اقتصاصه، الجداول
    الواسعة (تمرير أفقي أو تحوّل لبطاقات على الموبايل)، النوافذ المنبثقة (modals) وحجمها على الشاشة
    الصغيرة، وأن لا يُصمَّم لجهاز ويُهمَل الآخر. عرض مكسور على أحد الجهازين = عيب احترافي يُسجَّل.
  - الازدواجية المفاهيمية: ميزتان تفعلان الشيء نفسه بأسماء مختلفة (مثل "الوحدات" مقابل "أنواع الخدمات")
    تربك المستخدم — وحّد أو افصل الأدوار بوضوح.

لكل اكتشاف: الحقول الخمسة (الحقل 4 هنا = أثر على الاحترافية/كفاءة العمل/ثقة المستخدم، لا الأثر المالي).
التصنيف: عادةً Medium/Low — لكن "ميزة وهمية تُعرض كأنها تعمل" = High (تُضلّل المستخدم وقد تُفقده عمله/ثقته)،
  و"تجميل UI بلا فرض خلفي لميزة أمنية" يُرفَع لـ Critical/High حسب الطور 4.
```

🔄 **بعد تسليم تقرير الاحترافية/UX:** بدّل إلى **Opus 4.8 · Max** للتقرير النهائي والحكم.

## 8) التقرير النهائي الموحّد

```
اجمع كل الأطوار في تقرير واحد:

## 9) انضباط المعالجة (Remediation) — حين تُكلَّف بالإصلاح لا التدقيق فقط

عند تجاوز التدقيق إلى الإصلاح الفعلي، التزم بهذا الانضباط (مستخلَص من جلسات إصلاح سابقة):

- **إصلاح واحد لكل commit**، برسالة تشرح السيناريو الواقعي والأثر المالي — لا "fix bug".
- **بوابة CLAUDE.md الإلزامية قبل كل دفع:** `tsc` نظيف + `next build` ناجح. البناء فشل
  سابقاً مرتين؛ لا دفع قبل بناء أخضر مؤكَّد بالقراءة.
- **أي تغيير DB يدخل `instrumentation.ts`** (المُطبِّق الفعلي) مع ملف drizzle مطابق للتوثيق —
  وإلا فالإصلاح خامل إنتاجياً (راجع الطور 0 بند 7).
- **لإصلاح حارس** (RLS/تزامن/قفل فترة): أعِد اختبار الحارس نفسه، وصرّح بصدق إن كان
  "مُطبَّقاً لكن لم يُختبر على DB حية".
- **بعد سلسلة الإصلاحات:** حدّث قسم "الحالة الحالية" في CLAUDE.md (ما نُفِّذ/ما أُجِّل) واكتب
  تقرير حصيلة في `docs/AUDIT-SESSION-<التاريخ>.md` بالهاشات.

### ملاحظة منهجية أخيرة
لماذا الموديل مباشرةً لا الوكلاء؟ لأن أخطاء الترابط المحاسبي (قيد يتوازن داخل دالة واحدة لكنه يختل عبر خطوتين متتاليتين) لا يكتشفها إلا عقل واحد يمسك الخيط كاملاً من العميل حتى journal_lines. الوكيل الفرعي يقرأ جزءاً ويُعيد ملخصاً، فيضيع السطر الذي تقوم عليه قاعدة الدليل. لذلك: اقرأ أنت، واحكم أنت، ونفّذ أنت — طوراً بطور، كل طور في جلسة نظيفة إن أمكن حتى لا يتلوّث السياق ويضعف عمق القراءة.
